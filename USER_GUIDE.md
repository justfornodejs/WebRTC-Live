# WebRTC Live 实时直播平台 — 使用说明书

---

## 一、系统概述

WebRTC Live 是一个基于 WebRTC + SRS 的浏览器端低延迟实时直播系统。  
支持通过浏览器进行 **屏幕共享推流** 或 **摄像头推流**，其他用户可在浏览器中实时观看，延迟通常在 **50-300ms**。

**典型使用场景：**
- 会议同屏：主讲人共享屏幕，参会者实时观看
- 教学直播：老师共享 PPT/操作演示，学生端观看
- 远程协助：共享操作画面给远端技术人员

---

## 二、访问地址

| 页面 | 地址 | 说明 |
|------|------|------|
| 首页 | `https://服务器IP:5000/` | 系统首页、架构介绍 |
| 推流页 | `https://服务器IP:5000/publish.html` | 发起推流（主播端） |
| 拉流页 | `https://服务器IP:5000/play.html` | 观看直播（观众端） |

> 将 `服务器IP` 替换为实际部署的局域网 IP，例如 `https://192.168.1.100:5000/`

## 三、首次使用（信任证书）

由于系统使用自签名 SSL 证书，**每台电脑首次访问时**浏览器会提示安全警告：

### Chrome / Edge
1. 打开 `https://服务器IP:5000`
2. 看到"您的连接不是私密连接"提示
3. 点击 **「高级」**
4. 点击 **「继续前往 xxx（不安全）」**
5. 完成，后续不再提示

### Firefox
1. 打开 `https://服务器IP:5000`
2. 看到"警告：面临潜在的安全风险"
3. 点击 **「高级」**
4. 点击 **「接受风险并继续」**

> 💡 此操作每台电脑只需做一次，之后浏览器会记住。

## 四、推流操作（主播端）

### 4.1 进入推流页面

打开 `https://服务器IP:5000/publish.html`

### 4.2 选择推流源

在右侧控制面板的「推流源」下拉框中选择：

| 推流源 | 说明 | 适用场景 |
|--------|------|----------|
| 🖥️ 屏幕共享 | 共享整个屏幕或单个窗口 | 演示 PPT、操作软件、远程协助 |
| 📷 摄像头 | 使用电脑摄像头画面 | 人像直播、视频会议 |
| 📁 视频文件 | 选择本地视频文件推流 | 播放预录制的教学视频 |

### 4.3 开始推流

1. 选好推流源后，点击 **「▶️ 开始推流」**
2. 如选择「屏幕共享」，系统会弹出共享选择窗口：
   - **整个屏幕**：共享整个显示器画面
   - **应用窗口**：只共享某个应用窗口（推荐）
   - **浏览器标签页**：共享某个浏览器标签
3. 选择后点击「共享」，推流开始
4. 左侧出现画面预览，左上角显示红色 **LIVE** 标记
5. 右侧「实时统计」显示分辨率、帧率、码率

### 4.4 停止推流

- 点击 **「⏹️ 停止推流」** 按钮
- 或者在浏览器/系统的共享提示条中点击「停止共享」

### 4.5 推流注意事项

- ⚠️ **不要关闭推流页面**，关闭即停止推流
- ⚠️ 推流期间切换窗口不影响屏幕共享（共享的是选定的屏幕/窗口）
- 💡 推荐使用 **Chrome** 或 **Edge** 浏览器，兼容性最好
- 💡 推荐选择「应用窗口」模式共享，避免暴露桌面隐私

## 五、拉流观看（观众端）

### 5.1 进入拉流页面

打开 `https://服务器IP:5000/play.html`

### 5.2 开始观看

1. 确认推流端已经在推流（重要！）
2. 点击 **「▶️ 开始拉流」**
3. 等待 2-5 秒，视频画面出现
4. 左上角显示 **LIVE** 标记，表示正在直播

### 5.3 停止观看

点击 **「⏹️ 停止播放」**

### 5.4 拉流注意事项

- ⚠️ **必须先有人推流**，拉流才能看到画面
- ⚠️ 如果拉流失败，请检查日志面板中的错误信息
- 💡 多个观众可以同时拉流观看，互不影响
- 💡 如果画面卡顿，可先停止再重新拉流

## 六、常见问题排查

### Q1：点击「开始推流」没反应 / 报错

| 原因 | 解决方法 |
|------|----------|
| 未信任证书 | 先访问 `https://服务器IP:5000` 信任证书 |
| 使用了 HTTP 访问 | 必须用 **HTTPS**（`https://`），否则浏览器禁止屏幕共享 |
| 浏览器不支持 | 使用 Chrome 88+ / Edge 88+ / Firefox 85+ |

### Q2：拉流页面一直黑屏

| 原因 | 解决方法 |
|------|----------|
| 推流端未推流 | 确认推流端已成功开始推流 |
| 推流端已断开 | 让推流端重新推流，拉流端重新拉流 |
| 防火墙阻止 | 确认服务器 UDP 8000 端口已放行 |

### Q3：日志显示「SRS 错误」或「信令请求失败」

```
解决步骤：
1. 检查 SRS 是否运行：  systemctl status srs
2. 检查 Flask 是否运行： systemctl status webrtc-live
3. 重启服务：
   systemctl restart srs
   systemctl restart webrtc-live
```

### Q4：画面延迟很高或卡顿

- 检查局域网带宽是否充足（1080p 推流约需 2-4 Mbps）
- 推流端降低分辨率：可改用 720p
- 确认推流端和服务器不在同一台机器上时网络稳定

### Q5：手机能用吗？

- **拉流观看**：✅ 手机浏览器支持（Chrome/Safari）
- **推流**：⚠️ 部分手机浏览器支持摄像头推流，不支持屏幕共享

## 七、多路流（高级用法）

默认推流地址是固定的一路流。如果需要**多个人同时推不同的流**（如多个教室），可以使用不同的流名称。

### 方法

修改 `.env` 文件或在代码中设置不同的 `streamurl`：

| 场景 | 流地址 |
|------|--------|
| 默认 | `webrtc://服务器IP:1985/live/livestream` |
| 教室1 | `webrtc://服务器IP:1985/live/classroom1` |
| 教室2 | `webrtc://服务器IP:1985/live/classroom2` |
| 会议A | `webrtc://服务器IP:1985/live/meetingA` |

> 路径格式：`webrtc://IP:1985/应用名/流名称`  
> `live` 是默认应用名，后面的名称可以自由定义。

推流端和拉流端使用**相同的流地址**即可匹配。

## 八、服务管理（运维）

### 查看服务状态
```bash
systemctl status srs            # SRS 流媒体服务器
systemctl status webrtc-live    # Flask Web 服务
```

### 重启服务
```bash
systemctl restart srs
systemctl restart webrtc-live
```

### 查看实时日志
```bash
journalctl -u srs -f            # SRS 日志
journalctl -u webrtc-live -f    # Flask 日志
```

### 检查端口占用
```bash
ss -tlnp | grep -E '5000|1985'    # TCP 端口
ss -ulnp | grep 8000              # UDP 端口
```

### 健康检查 API
```bash
curl -k https://localhost:5000/api/health
```
返回 JSON 包含 SRS 连接状态。

## 九、网络端口一览

| 端口 | 协议 | 服务 | 是否必须对客户端开放 |
|------|------|------|----------------------|
| 5000 | TCP | Flask HTTPS（页面+API） | ✅ 是 |
| 8000 | UDP | WebRTC 媒体传输 | ✅ 是 |
| 1985 | TCP | SRS HTTP API | ❌ 仅内部使用 |
| 1935 | TCP | RTMP（OBS 推流） | ❌ 可选 |
